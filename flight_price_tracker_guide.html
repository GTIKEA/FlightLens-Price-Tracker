What this project is

The simple idea

Flight prices change every few hours. You write a Python script that visits a website, reads the price of a flight (say, Bangalore → Delhi), and saves it to a database. You do this every day for a few weeks.

After collecting data, you write SQL queries to answer questions like: Which day of the week is cheapest to fly? How far in advance should I book? Do prices spike on weekends?

That's it. The result is a real dataset, real insights, and 4–5 impressive SQL queries that prove you can think like a data analyst.

What you will build

A MySQL database with 4 tables, a Python scraper that runs daily, and a set of queries using window functions, CTEs, and GROUP BY that produce business insights.

Why it stands out

Most SQL projects use downloaded CSV files. This project generates its own real data over time, which means you can say "I collected 2,000+ price snapshots over 3 weeks" — that is unique.

💡
Time estimate: Set up takes 1–2 days. Then let it collect data for 2–3 weeks while you do other things. Come back and write the queries. Total active effort: ~15–20 hours.

Setting up the database

Open MySQL Workbench and run this SQL

Copy the entire block below and run it. It creates the database and all 4 tables.

📄 setup_database.sql
-- Step 1: Create the database CREATE DATABASE IF NOT EXISTS flight_tracker; USE flight_tracker; -- Table 1: Routes (BLR→DEL, BOM→HYD, etc.) CREATE TABLE routes ( route_id INT AUTO_INCREMENT PRIMARY KEY, origin CHAR(3) NOT NULL, -- airport code, e.g. 'BLR' destination CHAR(3) NOT NULL, -- airport code, e.g. 'DEL' distance_km INT, UNIQUE(origin, destination) ); -- Table 2: Flights (each specific flight on a date) CREATE TABLE flights ( flight_id INT AUTO_INCREMENT PRIMARY KEY, route_id INT, airline VARCHAR(50), flight_number VARCHAR(10), departure_date DATE, departure_time TIME, duration_min INT, stops TINYINT DEFAULT 0, -- 0 = direct, 1 = 1 stop FOREIGN KEY(route_id) REFERENCES routes(route_id) ); -- Table 3: Price snapshots (the main table — price at each point in time) CREATE TABLE price_snapshots ( snapshot_id INT AUTO_INCREMENT PRIMARY KEY, flight_id INT, scraped_at DATETIME DEFAULT CURRENT_TIMESTAMP, price_inr DECIMAL(10,2), seats_left TINYINT, -- urgency signal from website class ENUM('economy','business') DEFAULT 'economy', FOREIGN KEY(flight_id) REFERENCES flights(flight_id) ); -- Table 4: Price alerts (bonus feature — track when price drops) CREATE TABLE price_alerts ( alert_id INT AUTO_INCREMENT PRIMARY KEY, route_id INT, threshold_inr DECIMAL(10,2), triggered_at DATETIME DEFAULT NULL, -- NULL means not triggered yet FOREIGN KEY(route_id) REFERENCES routes(route_id) ); -- Insert your routes INSERT INTO routes (origin, destination, distance_km) VALUES ('BLR', 'DEL', 1740), ('BLR', 'BOM', 840), ('DEL', 'CCU', 1305), ('BOM', 'HYD', 620);
✅
Check it worked: In MySQL Workbench, run SHOW TABLES; — you should see 4 tables listed.

Writing the scraper

How scraping works (simply)

Your Python script pretends to be a browser. It requests the MakeMyTrip search page, gets the HTML back, finds the price numbers in the HTML, and saves them to your database.

The tricky part is that many travel sites block scrapers. The recommended approach for beginners is to use a public API alternative or to scrape Google Flights via SerpAPI (has a free tier).

⚠️
Beginner-friendly alternative: Use the Aviationstack API (free plan: 500 calls/month) or SerpAPI (100 free searches/month) to get real flight data without fighting anti-scraping systems. Sign up at aviationstack.com or serpapi.com and get a free API key.
📄 scraper.py
import requests import mysql.connector from datetime import datetime, date, timedelta import time import random # ── CONFIG ──────────────────────────────────────────────────────── DB_CONFIG = { 'host': 'localhost', 'user': 'root', 'password': 'your_password_here', # change this 'database': 'flight_tracker' } SERPAPI_KEY = 'your_serpapi_key_here' # get free key at serpapi.com # Routes to track: (route_id, origin, destination) ROUTES = [ (1, 'BLR', 'DEL'), (2, 'BLR', 'BOM'), (3, 'DEL', 'CCU'), ] # ── DATABASE CONNECTION ─────────────────────────────────────────── def get_connection(): return mysql.connector.connect(**DB_CONFIG) # ── GET FLIGHT DATA FROM SERPAPI ────────────────────────────────── def fetch_flights(origin, destination, travel_date): """Call SerpAPI Google Flights endpoint and return list of flights.""" url = "https://serpapi.com/search" params = { "engine": "google_flights", "departure_id": origin, "arrival_id": destination, "outbound_date": travel_date.strftime("%Y-%m-%d"), "currency": "INR", "hl": "en", "api_key": SERPAPI_KEY, } response = requests.get(url, params=params, timeout=15) data = response.json() results = [] for flight in data.get("best_flights", []): for leg in flight.get("flights", []): results.append({ "airline": leg.get("airline", ""), "flight_number": leg.get("flight_number", ""), "departure_time": leg.get("departure_airport", {}).get("time", "00:00"), "duration_min": leg.get("duration", 0), "price_inr": flight.get("price", 0), }) return results # ── SAVE TO DATABASE ────────────────────────────────────────────── def save_snapshot(conn, route_id, travel_date, flight_data): cursor = conn.cursor() # 1. Insert into flights (or get existing flight_id) cursor.execute(""" INSERT IGNORE INTO flights (route_id, airline, flight_number, departure_date, departure_time, duration_min) VALUES (%s, %s, %s, %s, %s, %s) """, ( route_id, flight_data['airline'], flight_data['flight_number'], travel_date, flight_data['departure_time'], flight_data['duration_min'] )) cursor.execute(""" SELECT flight_id FROM flights WHERE route_id=%s AND flight_number=%s AND departure_date=%s """, (route_id, flight_data['flight_number'], travel_date)) row = cursor.fetchone() if not row: return flight_id = row[0] # 2. Insert price snapshot cursor.execute(""" INSERT INTO price_snapshots (flight_id, price_inr) VALUES (%s, %s) """, (flight_id, flight_data['price_inr'])) conn.commit() cursor.close() # ── MAIN LOOP ───────────────────────────────────────────────────── def main(): conn = get_connection() print(f"[{datetime.now()}] Starting scrape...") for route_id, origin, dest in ROUTES: # Check flights 7, 14, 21, 30 days from today for days_ahead in [7, 14, 21, 30]: travel_date = date.today() + timedelta(days=days_ahead) try: flights = fetch_flights(origin, dest, travel_date) for f in flights[:5]: # save top 5 flights per route save_snapshot(conn, route_id, travel_date, f) print(f" ✓ {origin}→{dest} ({days_ahead}d) — {len(flights)} flights saved") except Exception as e: print(f" ✗ {origin}→{dest} error: {e}") time.sleep(random.uniform(1.5, 3)) # be polite, don't hammer the API conn.close() print("Done!") if __name__ == "__main__": main()
✅
Test it: Replace your credentials and API key, then run python scraper.py. Check MySQL Workbench: run SELECT * FROM price_snapshots LIMIT 10; to see your first rows.

Writing the SQL queries

These are the 5 queries you will put in your README and talk about in interviews. Each one answers a real question. Write them in MySQL Workbench and save the screenshots.

Query 1 — Cheapest day to fly (GROUP BY + RANK)

Answers: "If I fly on a Tuesday vs a Friday, how much do I save?"

SELECT r.origin, r.destination, DAYNAME(f.departure_date) AS day_of_week, ROUND(AVG(p.price_inr), 0) AS avg_price_inr, RANK() OVER ( PARTITION BY r.route_id ORDER BY AVG(p.price_inr) ASC ) AS cheapest_rank FROM price_snapshots p JOIN flights f ON p.flight_id = f.flight_id JOIN routes r ON f.route_id = r.route_id GROUP BY r.route_id, DAYOFWEEK(f.departure_date) ORDER BY r.route_id, cheapest_rank;

Query 2 — Best time to book in advance (CTE + DATEDIFF)

Answers: "Does booking 3 weeks early actually save money vs booking last minute?"

WITH booking_gap AS ( SELECT f.route_id, DATEDIFF(f.departure_date, p.scraped_at) AS days_ahead, p.price_inr FROM price_snapshots p JOIN flights f ON p.flight_id = f.flight_id WHERE DATEDIFF(f.departure_date, p.scraped_at) BETWEEN 1 AND 90 ) SELECT route_id, FLOOR(days_ahead / 7) * 7 AS week_bucket, ROUND(AVG(price_inr), 0) AS avg_price, COUNT(*) AS data_points FROM booking_gap GROUP BY route_id, week_bucket ORDER BY route_id, week_bucket;

Query 3 — 7-day rolling average price (Window function)

Answers: "Is the price trending up or down this week?"

SELECT f.route_id, DATE(p.scraped_at) AS snapshot_date, ROUND(AVG(p.price_inr), 0) AS daily_avg, ROUND( AVG(AVG(p.price_inr)) OVER ( PARTITION BY f.route_id ORDER BY DATE(p.scraped_at) ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ), 0) AS rolling_7d_avg FROM price_snapshots p JOIN flights f ON p.flight_id = f.flight_id GROUP BY f.route_id, snapshot_date ORDER BY f.route_id, snapshot_date;

Query 4 — Sudden price spike detection (Self JOIN)

Answers: "Which flights jumped more than 20% since yesterday?"

SELECT p.flight_id, p.scraped_at AS now, prev.scraped_at AS previous, p.price_inr AS current_price, prev.price_inr AS previous_price, ROUND((p.price_inr - prev.price_inr) / prev.price_inr * 100, 1) AS pct_change FROM price_snapshots p JOIN price_snapshots prev ON prev.flight_id = p.flight_id AND prev.scraped_at = ( SELECT MAX(scraped_at) FROM price_snapshots WHERE flight_id = p.flight_id AND scraped_at < p.scraped_at ) WHERE p.price_inr > prev.price_inr * 1.20 ORDER BY pct_change DESC;

Query 5 — Direct vs connecting flight price gap

Answers: "How much extra does the direct flight cost vs a 1-stop?"

SELECT r.origin, r.destination, f.stops, COUNT(*) AS flight_count, ROUND(AVG(p.price_inr), 0) AS avg_price, ROUND(MIN(p.price_inr), 0) AS min_price, ROUND(MAX(p.price_inr), 0) AS max_price FROM price_snapshots p JOIN flights f ON p.flight_id = f.flight_id JOIN routes r ON f.route_id = r.route_id GROUP BY r.route_id, f.stops ORDER BY r.route_id, f.stops;

Running it automatically every day

You do not want to run the script manually every day. Set up a scheduler to run it for you automatically.

On Windows — Task Scheduler

1

Open Task Scheduler

Search "Task Scheduler" in the Start menu

2

Create Basic Task

Click "Create Basic Task" in the right panel

3

Set trigger to Daily

Choose "Daily" → set time to 9:00 AM

4

Set action

Action: Start a program → browse to python.exe → add argument: full path to scraper.py

On Mac/Linux — Cron Job

Open your terminal and type crontab -e, then add this line:

0 9 * * * /usr/bin/python3 /path/to/scraper.py >> /tmp/flight_log.txt 2>&1

This runs the scraper at 9 AM every day and saves output to a log file so you can check it ran correctly.

✅
Check progress: After 3 days, run SELECT COUNT(*) FROM price_snapshots; in MySQL Workbench. You should see at least 30–50 rows. After 3 weeks you will have 300–600+ rows — enough to run all your queries with meaningful results.

Interview Q&A prep

These are the exact questions you will be asked. Read the answers, understand them, and put them in your own words.

Questions you will be asked

Q: Walk me through your project.
I built a system that tracks flight prices daily. A Python script calls the SerpAPI to get current prices for 4 routes, stores them in a MySQL database, and I let it run for 3 weeks. I then wrote SQL queries to find patterns — like which day is cheapest and how booking time affects price. The most interesting finding was that BLR→DEL prices on Fridays are 41% higher than Tuesdays.
Q: Why did you use a RANK() window function in Query 1?
I wanted to rank the days of the week by average price within each route separately. If I ranked globally, the ranking would mix up routes with different base prices. PARTITION BY route_id means I get rank 1 (cheapest day) per route independently.
Q: What is a CTE and why did you use one in Query 2?
A CTE (Common Table Expression) is like a temporary named result set. I used one to first calculate the days_ahead value for each snapshot, then group by week buckets in a second step. Without the CTE I would have to nest subqueries which is harder to read and debug.
Q: Why is your schema normalised into 4 tables instead of one big table?
Because routes and flight details do not change every time I scrape a price. If I stored everything in one table, I would repeat "BLR→DEL, IndiGo, 6E-201" thousands of times. Normalisation avoids that duplication, makes updates easier, and keeps the price_snapshots table small and fast to query.
Q: What would you do differently if you did this project again?
I would add an index on (flight_id, scraped_at) in price_snapshots, because Query 4 does a self-join on that table and it becomes slow as data grows. I would also collect the seats_left column more consistently — I dropped it sometimes when the API did not return it, which made that column unreliable for analysis.
Q: Is this data reliable for actual travel decisions?
It is directionally reliable but not perfectly accurate — prices change by the minute and the API has some latency. The value is the trend analysis over weeks, not predicting the price at any single moment. For a resume project, the goal is showing I can collect, model, and query time-series data, which it does.
